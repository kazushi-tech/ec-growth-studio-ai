# Insight Studio vs EC Growth Studio AI — 役割と売り方の違い

> 上司・社内メンバー・初回商談で **「この2つは何が違うの？」「なぜ別プロダクトなの？」**
> と聞かれた時に、30秒〜3分で答えられる状態を作るためのドキュメント。
>
> 結論: **どちらが上位／下位ではなく、ターゲットも商売モデルも違う別プロダクト**。
> ただし血統は近く、**Insight Studio で実際に動いている BigQuery / AI 基盤を、
> EC Growth Studio AI の Phase 3 / 4 で EC月次運用ループに転用する** という接続関係にある。

---

## 0. 一言での違い

| | 一言で |
| --- | --- |
| **Insight Studio** | 広告代理店・運用チーム向けの **「LP / 競合 / クリエイティブを実データで深掘りする分析スタジオ」**（実運用中） |
| **EC Growth Studio AI** | EC事業者向けの **「売上要因分解→施策→月次レポートを毎月回す運用ループの司令塔」**（プロトタイプ） |

- Insight Studio は **「分析者向けのワークベンチ」**。
- EC Growth Studio AI は **「EC担当者向けの月次運用画面」**。
- 前者は **案件スポット深掘り**、後者は **月次BPaaS伴走**。

## 1. 対象ユーザーの違い

| 軸 | Insight Studio | EC Growth Studio AI |
| --- | --- | --- |
| 主な利用者 | 広告代理店 / インハウス広告運用チーム / クリエイティブチーム | EC事業者の経営者・販促責任者・店長・運用担当（社内＋伴走するBPaaS担当） |
| 利用シーン | 案件アサイン時のLP診断・競合調査・広告深掘り・クリエイティブ評価 | 毎月の売上レビュー・施策決定・月次顧客レポート提出 |
| 契約形態 | スポット案件分析 / 月額利用 | **初回診断 → 月次運用 → BPaaS伴走** の3段階 |
| ターゲット規模 | 広告代理店の案件〜大手クライアント | 月商 100万〜5,000万円規模の EC / D2C 事業者 |
| 利用粒度 | 「案件単位」（LP1本・キャンペーン1本） | 「店舗単位」「月次単位」 |

要点: **ペルソナがそもそも別**。代理店アナリストと EC 担当者は、
KPI の見方も時間軸も意思決定プロセスも違う。1つの UI で両方を満たそうとすると、
両方のユーザーにとって浅い画面になる。

## 2. 扱うデータの違い

| 軸 | Insight Studio | EC Growth Studio AI |
| --- | --- | --- |
| 入力 | LP の URL / 広告ライブラリ / GA4（実接続）/ BigQuery（実接続）/ Claude API キー | **注文 CSV / GA4 CSV / 広告 CSV**（手元のエクスポート） |
| 接続方式 | Render 上の FastAPI バックエンド経由 + BigQuery 直クエリ + Claude API | **CSV-first / API-later**。ブラウザ内 `localStorage` のみ |
| 永続化 | PostgreSQL + BigQuery + 案件単位レポート履歴 | なし（再読込で復元される直前取込のみ） |
| AI連携 | **Claude API 実接続（prompt cache 必須運用）** | **静的サンプル**（Phase 4 で Anthropic SDK 接続予定） |
| 認証 | 案件ベース + RBAC（管理者 / 案件ユーザー）+ Ads Insights パスワード認証 | **なし**（プロトタイプ段階で意図的に未実装） |

要点: **Insight Studio は「実データで動く製品」**、
**EC Growth Studio AI は「画面とワークフローの設計検証フェーズ」**。
段階が違うだけでなく、**入手しやすいデータの形が違う**:
代理店は LP URL や広告ライブラリを持つが、EC事業者は CSV を持つ。

## 3. 主要ユースケースの違い

### Insight Studio の主役シーン

1. **LP比較分析（`/compare`）** — 自社LPと競合LPを並べて、構成・訴求・ファネルの差を Claude が分解
2. **競合発見（`/discovery`）** — キーワード／カテゴリから競合候補を発見、広告・LP・市場ポジションを把握
3. **クリエイティブ診断（`/creative-review`）** — 画像広告 / バナー / 動画キャプチャに対する Claude レビュー
4. **広告分析（`/ads/graphs`）** — BigQuery × GA4 × 広告データで CTR / CPA / ROAS の深掘り
5. **AIエクスプローラー（`/ads/ai`）** — 広告データに対する自由質問 → Claude 応答 + チャート生成

### EC Growth Studio AI の主役シーン

1. **データ取込・連携（`/app/data-import`）** — 注文CSV / GA4 CSV / 広告CSV を投入、CSV-first 思想
2. **売上要因分析（`/app/revenue-analysis`）** — 売上 = セッション × CVR × AOV を **連鎖法で要因分解**、主因候補と次の一手を出す
3. **ダッシュボード（`/app`）** — 月次KPI + AI月次診断サマリー + 優先施策 + 商品別判断
4. **AI考察レポート（`/app/ai-report`）** — 課題4分解（商品ページ / 広告 / CRM / 在庫）+ AI改善提案
5. **施策ボード（`/app/action-board`）** — カンバン5列で **担当・期限・効果検証** まで管理
6. **月次レポート（`/app/monthly-report`）** — 顧客提出用レポートのレイアウト設計

要点: **Insight Studio は「分析と発見」が主役**、
**EC Growth Studio AI は「実行管理と月次提出」が主役**。
EC Growth Studio AI で「分析」だけを切り出しても、**運用ループの一部にしかならない**。

## 4. デモで見せるべき画面の違い

### Insight Studio デモ（実運用の説得力で押す）

- **`/compare`** で LP1 と LP2 を実際に投入し、Claude のレビューが走る
- **`/discovery`** で実競合を発見、生きた広告ライブラリと並べる
- **`/ads/graphs`** で BigQuery × GA4 のチャートが描画される
- **`/ads/ai`** で自由質問にチャート付きの応答が返る

→ **「実APIで本物のClaudeが動いている」「BigQueryに本当に繋がっている」** が刺さる。

### EC Growth Studio AI デモ（運用ループの設計の説得力で押す）

- **`/`（LP）** で「データ分散・優先順位がつかない・実行されない・報告で終わる」の4課題提起
- **`/app/data-import`** で注文 / GA4 / 広告の3CSVをドラッグ&ドロップで実演
- **`/app/revenue-analysis`** で売上要因分解（実値ラベル / 推定ラベル切替）を見せる
- **`/app/action-board`** でカンバンと効果検証パネル
- **`/app/monthly-report`** でレポート構成プレビュー

→ **「CSVだけで月次運用が回る」「AIで終わらせない、実行と月次レポートまで一貫」** が刺さる。

詳細な台本: [demo-script.md](demo-script.md) / 動画版構成: [demo-video-storyboard.md](demo-video-storyboard.md)

## 5. 収益化・商談での売り方の違い

| 軸 | Insight Studio | EC Growth Studio AI |
| --- | --- | --- |
| 売り方 | 案件分析サービス + ツール提供 | **月次BPaaS（伴走 + ツール）** |
| 課金単位 | 案件 / 月額 / API利用量（Claude キーは顧客持ち or こちら持ち） | **初回診断 / 月次運用 / 伴走** の3段階個別提案 |
| 提案先 | 広告代理店の経営層 / 運用責任者 | EC事業者の経営者 / 販促責任者 |
| 利益源 | 分析・調査の代行・自動化 + ツール販売 | 月次運用代行 + 効果検証 + 月次レポート提出（人＋ツール） |
| LTV の作り方 | 案件継続 / 月額 SaaS | **月次運用契約**（解約しないと止まらない） |
| 競合 | DataPortal / Looker / 自社制作分析ダッシュボード | **「分析だけ」のレポートツール全般 + コンサル単独**（実行管理ができない） |

要点: **EC Growth Studio AI は「ツール売り」ではなく「月次運用ループ売り」**。
他社レポートツールと並べた時の差別化は **「BPaaS伴走 + 施策実行管理 + 月次レポート出力まで一貫」**。
Insight Studio は **「実データで動く分析スタジオ」** が売り。

## 6. 役割の違い（上位／下位ではない）

```
┌──────────────────────┐         ┌──────────────────────┐
│  Insight Studio      │         │  EC Growth Studio AI │
│  ・代理店・運用チーム │         │  ・EC事業者          │
│  ・LP / 競合 / 広告  │         │  ・売上 / 施策 / 月次 │
│  ・分析・発見・診断  │         │  ・運用・実行・報告   │
│  ・実API / 実AI      │         │  ・CSV-first         │
└──────────────────────┘         └──────────────────────┘
                ＼                      ／
                  ＼                  ／
        ┌──────────────────────────────┐
        │  共通の血統:                   │
        │  ・Anthropic / Claude 重視     │
        │  ・BigQuery + GA4 への現実主義 │
        │  ・CSV → 実データ の段階主義   │
        │  ・人間レビュー前提の AI 設計   │
        └──────────────────────────────┘
```

- **どちらが上位／下位でもない**。同じ会社の中で **異なる顧客層に異なる商品** を出している関係。
- **Insight Studio は実運用、EC Growth Studio AI はプロトタイプ** という段階の違いはあるが、
  これは **EC Growth が劣っているのではなく、商談で優先項目を決めてから本格実装に入る方針** の結果。
- **将来の接続**: Insight Studio で構築済みの BigQuery 読み取り基盤・Claude prompt cache 運用を、
  Phase 3 / Phase 4 で EC Growth Studio AI に **そのまま転用** する想定（→ 8章）。

## 7. Insight Studio の実績を EC Growth Studio AI の説得材料として使う言い方

商談・上司デモで **「EC Growth Studio AI はまだプロトタイプですが…」** と言う必要はない。
代わりに、**Insight Studio の実績を裏付けに** こう言う:

### 7-1. 30秒版

> 「**Insight Studio という別プロダクトで、Claude API + BigQuery + GA4 の実接続は
> すでに本番稼働しています**。EC Growth Studio AI はその技術基盤を **EC事業者の月次運用ループ** に
> 転用する設計です。今お見せしているのは **画面とワークフローの確定版** で、API接続と AI 実呼び出しは
> Phase 3 / 4 で Insight Studio 側から移植します。」

### 7-2. 想定インパクト

- **「実装力の不安」を消す**: Insight Studio が動いている実績で、AI / BigQuery 連携の技術リスクは取れている
- **「プロトタイプを買わされる不安」を消す**: 画面は確定版、技術は別リポで実証済み、というメッセージにできる
- **「なぜ別プロダクトなのか」をポジティブに語れる**: ターゲットが違うから別UIが正解、という説明が成立する

### 7-3. 言ってはいけないこと

- ❌ 「Insight Studio の EC 版です」 → **嘘になる**。ターゲットも UX も違う
- ❌ 「Insight Studio をもっと良くしたものです」 → **上下関係を作ると Insight Studio の価値を下げる**
- ❌ 「Insight Studio と統合する予定です」 → **統合する予定はない**。**技術の転用** であって **製品の統合ではない**
- ✅ 「**別プロダクト**です。技術基盤の一部は Insight Studio から転用します」

## 8. 上司に聞かれそうな質問と回答

### Q1. 「Insight Studio と EC Growth Studio AI、何が違うの？」（30秒）

A. **ターゲットと商売モデルが違う別プロダクト**です。Insight Studio は広告代理店・運用チーム向けの
LP / 競合 / クリエイティブを **実データで深掘りする分析スタジオ**。EC Growth Studio AI は
EC事業者向けの **月次の売上改善ループを回す BPaaS** プロトタイプです。前者は分析、後者は運用。
**どちらが上位／下位ではなく、それぞれの顧客層に対する別の商品**です。

### Q2. 「なぜ1つに統合しないの？」

A. **代理店アナリストと EC 担当者は、KPI の見方も時間軸も意思決定プロセスも違う**ため。
1画面に詰めると両方のユーザーに対して浅い画面になります。
ただし **技術基盤は共通化** していて、Phase 3 で Insight Studio の BigQuery 読み取り部分を
EC Growth Studio AI 側に転用する設計です。

### Q3. 「Insight Studio で動いてる Claude / BigQuery 連携を EC Growth Studio AI でなぜ動かさないの？」

A. **商談で優先項目を確定させてから本格実装に入る方針**だからです。現MVPは
**画面構造とワークフロー検証** が目的で、API接続 / AI実接続 / 永続化 / 認証 / PDF出力は
意図的に Phase 3 / 4 に分けています。**Insight Studio 側で実装済みの基盤があるので、
技術リスクは取れています**。

### Q4. 「カニバらないの？同じ会社で2つ売って大丈夫？」

A. **ターゲットが違うのでカニバりません**。Insight Studio は代理店向け、EC Growth Studio AI は
EC事業者向け。むしろ **代理店経由で EC 事業者に EC Growth Studio AI を案内する逆流線** も成立します
（代理店の月次運用代行ツールとして使ってもらう）。

### Q5. 「Insight Studio は完成しているのに、なぜ EC Growth Studio AI を新しく作ったの？」

A. **EC事業者の月次運用ループは、Insight Studio の延長では作れない** からです。Insight Studio は
案件スポット分析の世界観で、案件選択 → 分析実行 → レポート の流れ。EC Growth Studio AI は
店舗単位 → 月次 → 売上要因分解 → 施策 → 効果検証 → 月次レポート というループ構造で、
**画面構造そのものが別物** です。同じ UI に乗せると、両方の世界観が薄まります。

### Q6. 「BigQuery に直接繋げないの？」

A. **Phase 3 で Insight Studio の BigQuery 読み取り基盤をそのまま転用予定**です。
ただし主価値は **「BigQuery接続できること」ではなく「売上変動の原因分解と次アクション化」**。
GA4 / BigQuery は **そのための入力チャネル** という位置づけにしています。

### Q7. 「AIは本物が動いてないのに、AI考察って言っていいの？」

A. **現MVPでは静的サンプルとして明示**しています。デモ中も「AIが生成する想定の例文です」と
必ず付け加える運用です。**Phase 4 で Anthropic SDK + prompt cache 必須**で実装予定で、
prompt cache の運用ノウハウは Insight Studio 側で確立済みです。

### Q8. 「EC Growth Studio AI はいつ売れる状態になるの？」

A. **CSVベースの月次BPaaS伴走サービスとしては、現MVPの状態でも商談・契約を始められます**。
お客様の手元の CSV をもとに、月次の売上要因分解 → 施策定義 → 月次レポート提出までは
人間 + 現UIで運用可能。**API実接続 / AI実接続 / 永続化 / 認証 / PDF出力は 6〜12ヶ月の段階実装**を想定。

### Q9. 「他社の EC 分析ツール（DataPortal / Lookerなど）と何が違うの？」

A. **DataPortal / Looker は分析者向けのワークベンチ** で、本プロダクトは EC 担当者向けの
**月次運用ループ画面**です。売上要因分解 → AI考察 → 施策ボード → 効果検証 → 月次レポートまでを
1つの導線で繋ぐ点と、**BPaaS伴走（人間レビュー / 月次会議）が前提に組み込まれている** 点が
他社にない差別化要素です。

### Q10. 「2つ作って、両方を維持できるの？開発リソース大丈夫？」

A. **開発リソースの観点でも別プロダクトが正解**です。Insight Studio は実APIフル実装で開発負荷が高く、
EC Growth Studio AI は静的MVPなのでフロントエンドだけで構造検証できる軽さがあります。
**Phase 3 で技術基盤を共通化** することで、二重実装にはなりません。

## 9. まとめ — 上司に必ず伝えてほしい3点

1. **どちらが上位／下位ではなく、ターゲットと商売モデルが違う別プロダクト**である
2. **Insight Studio の実APIで動いている Claude / BigQuery 基盤** は、
   EC Growth Studio AI の **Phase 3 / 4 で技術転用される設計** であり、技術リスクは取れている
3. EC Growth Studio AI の主価値は **「GA4 / BigQuery 接続できること」ではなく
   「売上変動の原因分解と月次運用ループの司令塔」**。CSV-first で世界観を固めた上で
   段階的に API / AI / 永続化を厚くする
