### GCP / 実API 接続状況の前置き（誤解されやすいので必ず最初に言う）

上司や初見の方が **「BigQuery / GA4 / Shopify と本物で繋がっているのか」** と勘違いすることが多い。
デモ前に必ず以下を1分以内で言い切ってから本編に入る。

> 「現MVPは **実 GCP / 実 BigQuery / 実 GA4 API / 実 広告 API / 実 AI API には未接続** です。
> 今日お見せするのは **CSVで動いている部分** と **BigQueryデモ Mode（接続後の見え方の再現）** の2つ。
> 本格接続には **会社側の GCP プロジェクト・請求アカウント・サービスアカウント** の準備が必要なので、
> 勝手には接続しません。**Phase 3 後続 PR で実 BigQuery 接続、Phase 4 で 実 AI生成**に進む計画です。」

補足として:

- **CSV取込（実値）**: 注文 / GA4 / 広告 CSV をブラウザに取込み、売上 / セッション / CVR / ROAS を実値で再計算。
- **BigQuery デモ Mode（再現）**: Dashboard 上部のトグル ON で `mode:"mock"` 応答が KPI に流れる。
  画面上には常に「**GCP未接続 / 実BigQuery接続ではありません**」のバッジを併記。
- **AI考察**: `/app/ai-report` は **静的サンプル文言**。Phase 4 で Anthropic SDK + prompt cache に接続予定。
- **Shopify / GA4 / 広告 API**: 一切叩いていない。「接続する」ボタンは押さない。

> 補足の言い回し: 「画面に並んでいる **実値 / デモ / 未接続 / 将来予定** のラベルが、
> どの数字が本当に動いていて、どこが Phase 3 / Phase 4 で繋がるかを毎画面で示しています。」

#### BigQueryデモを上司デモで見せる場合の前提（重要）

BigQueryデモ Mode は **Vercel の `BQ_MOCK_MODE=true` が設定された Preview URL** でだけ動く。
Production（`https://ec-growth-studio-ai.vercel.app`）には **`BQ_MOCK_MODE` を入れない方針** のため、
Production で Dashboard 上部のトグルを ON にしても `/api/bq/orders-daily` は
`501 / NOT_IMPLEMENTED` を返して **安全停止** し、KPIは CSV / サンプル値にフォールバックする。

| 環境 | `BQ_MOCK_MODE` | BigQueryデモ トグル ON 時の挙動 |
| --- | --- | --- |
| Preview（デモ用） | `true` | `mode:"mock"` の固定 response が KPI に流れる（接続後の見え方の再現） |
| Production | 未設定 | `/api/bq/orders-daily` が `501 / NOT_IMPLEMENTED` で安全停止し、CSV/サンプル値にフォールバック |
| Production（実 GCP 接続後） | 未設定（`BQ_MOCK_MODE` は使わない） | Phase 3 後続 PR で実装予定。本PRでは実装しない |

つまり **「BigQueryデモを上司に見せる」ときは、必ず `BQ_MOCK_MODE=true` の Preview URL を開いてからデモする**。
Production をそのまま開いてトグルを ON にしても何も起きない（誤って実GCPに繋がない設計）。

> Production でトグルを ON にした場合、画面には赤いエラーではなく **「BigQueryデモは Preview 環境専用です」** という
> amber 系の安全停止案内が表示される（KPIは CSV / サンプル値のまま）。これは故障ではなく、未接続状態を安全に見せる UI である。

  - 「**ペタビット EC支援（伴走 / 代行）** の月次成果物を、AI診断 × 人間レビュー × 月次運用ループ の **統一フォーマット** に集約できます。」

## 2.9. Insight Studio との差別化セクション（聞かれたら必ずこれを言う）

> 詳細は [insight-studio-vs-ec-growth-studio.md](insight-studio-vs-ec-growth-studio.md) を参照。
> ここはデモ中に **「Insight Studio と何が違うの？」** と聞かれた瞬間に取り出す **30秒〜90秒の即答テンプレ**。

### 2-9-1. 30秒回答（万能・最初に出す1枚）

> 「**ターゲットと商売モデルが違う別プロダクト**です。Insight Studio は広告代理店・運用チーム向けの
> LP / 競合 / クリエイティブを実データで深掘りする **分析スタジオ**（実APIで本番稼働中）。
> EC Growth Studio AI は EC事業者向けの **月次の売上改善ループを回す BPaaS** プロトタイプです。
> **前者は分析、後者は運用。どちらが上位／下位ではなく、それぞれの顧客層に対する別の商品**です。」

### 2-9-2. 90秒回答（経営層・投資家・技術が気になる相手向け）

3点を順に話す:

1. **ターゲットの違い** — 「Insight Studio は **代理店アナリスト** 向け、EC Growth Studio AI は **EC事業者の販促責任者** 向け。
   KPI の見方も時間軸も意思決定プロセスも違うので、1画面に詰めると両方が浅くなります。」
2. **データ・接続の違い** — 「Insight Studio は **Claude API + BigQuery + GA4 の実接続** で本番稼働。
   EC Growth Studio AI は **CSV-first**（注文 / GA4 / 広告のCSVをブラウザ内localStorageで処理）で、
   実APIは Phase 3 / 4 で接続予定。」
3. **技術リスクは取れている** — 「Phase 3 で **Insight Studio 側の BigQuery 読み取り基盤を転用**、
   Phase 4 で **prompt cache 必須運用ノウハウをそのまま流用** する設計です。
   なので EC Growth Studio AI が `画面はあるけど中身がない` という状況にはなりません。」

### 2-9-3. 「Insight Studio で実際に動いている GA4 / BigQuery 連携を、EC Growth Studio ではどう発展させるのか？」

3段階で答える:

| Phase | 何を移植するか | EC月次運用ループでの効き方 |
| --- | --- | --- |
| **Phase 3（次）** | Insight Studio の **BigQuery 読み取り基盤** | 顧客が GA4 BigQuery Export を有効化していれば、`/app/revenue-analysis` の **セッション × CVR を自動更新化**。CSV取込が **不要に**なる |
| **Phase 4（その先）** | Insight Studio の **Anthropic SDK + prompt cache 運用** | `/app/ai-report` と **主因候補ヘッドライン** を実AI生成に置き換え。**月次の生成コストを prompt cache で抑える** |
| **Phase 4 後半** | Insight Studio の **案件ベース認証 + RBAC** | EC事業者単位 / 店舗単位のマルチテナント化。**Supabase + Edge Functions と組み合わせて永続化** |

要点: 「Insight Studio で実証済みの基盤を、**EC月次運用の文脈で組み直す**。
ゼロから作るのではなく、**動いている技術を別ドメインに転用する**」

### 2-9-4. EC Growth Studio AI の現在地と将来像（聞かれたら必ず3レイヤーで答える）

| | 現在地 | 次（〜3ヶ月） | 将来（6〜12ヶ月） |
| --- | --- | --- | --- |
| **状態** | CSV-first MVP プロトタイプ | 上司デモ / 商談導線 / BPaaS伴走サービス開始 | API / AI 実接続フル運用 |
| **データ** | 注文 / GA4 / 広告 CSV + 静的サンプル | 同上 + 実顧客のCSVで月次運用代行 | BigQuery / GA4 Data API / Shopify Admin API / 広告API |
| **AI** | 静的サンプル文 | 同上（AI生成テキストは BPaaS 担当が代行作成） | Anthropic SDK + prompt cache 必須 |
| **永続化** | ブラウザ内 localStorage | 同上 | Supabase + Edge Functions |
| **認証** | なし | なし（手元運用） | Clerk / Supabase Auth + 組織管理 + 操作ログ |
| **出力** | レイアウト設計のみ | 月次レポートを担当者が手動作成（UI流用） | PDF / PowerPoint / Google Docs 自動生成 |
| **何を売るか** | (まだ売らない) | **初回診断 + 月次BPaaS伴走サービス**（人＋現UI） | **SaaS + BPaaS** のハイブリッド契約 |

**伝え方のコツ**: 「**現状でも BPaaS伴走サービスとしては開始可能**。実APIが入る Phase 3 / 4 を待たなくても、
顧客のCSVをもとに月次の売上要因分解 → 施策定義 → 月次レポート提出までは **人＋現UIで運用できる**」
と必ず付け加える。「動かないMVPを売る」ではなく「**運用代行を売る**」と言い換えるのが肝。

## 3. 主要メッセージ（毎回どこかで必ず言う3つ）

### A. CSV-first / API-later — 「**CSVだけで月次運用を始められます**」

- 「Shopify Admin API / GA4 Data API / BigQuery / Google広告 API / Meta広告 API は **任意**。」
- 「導入初月は手元のCSVだけで売上要因分解 → AI診断 → 施策ボード → 月次レポートまで回せます。」
- 「APIは将来的に **読み取り専用** で順次追加。認証情報やAPIキーはまだ扱いません。」

### B. AIで終わらせない — 「**AI診断 × 人間レビュー × 月次運用ループ**」

- 「AIは『出発点』。最終判断は担当者が担い、施策ボードに **担当・期限・効果検証** を持たせます。」
- 「他社のAIレポートツールとの違いは **施策実行管理と月次レポート出力まで一貫している** ことです。」

### C. 月次BPaaSとして売れる — 「**伴走 + ツール** で初回診断・月次運用・効果検証まで提供**」

- 「BPaaS伴走で **AI生成内容のレビュー** と **月次会議** を担当します。」
- 「価格は『初回診断 / 月次運用 / BPaaS伴走』の **3段階個別提案**。スポット契約も可能。」

## 4. 想定質問と回答

### Q1. 「これは本物のAIが動いているんですか？」

A. **現時点ではAI生成テキストは静的な参考文** です。プロトタイプの目的は **画面構造とワークフローの検証** で、AI接続は Phase 4 で Anthropic SDK 経由で実装予定です。**prompt cache を必須**で設計します。AI生成 → 人間レビュー → 施策化 → 効果検証の流れは UI 上で再現済みです。

### Q2. 「Shopify とは APIで繋がっているんですか？」

A. **現MVPでは未接続** です。**CSV-first / API-later** の方針で、注文 / GA4 / 広告 の CSV を取込んで売上要因分解 → AI診断まで実値化する所まで実装済み。Shopify Admin API / GA4 Data API / BigQuery / 広告API は **読み取り専用** で Phase 3 以降に順次追加する計画です。

### Q3. 「データはどこに保存されているんですか？」

A. **ブラウザ内 localStorage のみ** で、外部送信は一切ありません。「サンプルデータで試す」を押すと取込状態は即時クリアされます。Phase 4 で Supabase + Edge Functions による永続化を予定していますが、現段階では認証情報やAPIキーを扱いません。

### Q4. 「CVR / ROAS の数字はどこから来ているんですか？」

A. CSVの取込状態によって **実値 / 推定値 / 未取込** を Pill ラベルで明示しています。

- 注文CSV取込済みなら **売上 / 注文数 / AOV が実値**
- GA4 CSV取込済みなら **セッション / CVR / チャネル別 / LP別 が実値**
- 広告CSV取込済みなら **広告費 / クリック / CPC / CVR / ROAS が実値**

それ以外は静的サンプル（=「推定」）として明示しています。前月数字は静的サンプルなので「推定」ラベル付きです。

### Q5. 「BigQuery に直接繋げられないんですか？」

A. **Phase 3 で BigQuery 読み取り接続を予定** しています。GA4 BigQuery Export を有効化していただいたうえで、サービスアカウント経由の **読み取り専用** 接続を想定。OAuth / GCP認証 / クエリ実行は **現MVPでは実装範囲外** です。GA4 / BigQuery 接続準備のチェックリスト（GCP プロジェクトID、dataset 名、必要権限）は `/app/data-import` 上で先に可視化しています。

### Q6. 「月次レポートは PDF で出ないんですか？」

A. **現MVPではレイアウト設計までで、実PDF出力は未実装** です。Phase 4 で `@react-pdf/renderer` または print stylesheet を試験実装する計画です。出力形式（PDF / PowerPoint / Google Slides / 共有リンク / Notion / Google Docs）は UI 上に並べていますが、すべて UI モックアップの段階です。

### Q7. 「他社のレポートツール（DataPortal / Lookerなど）との違いは？」

A. **本プロダクトは『分析画面』ではなく『判断と実行管理の画面』** です。DataPortal / Looker は分析者向けのワークベンチですが、本プロダクトはEC担当者向けの月次運用ループに特化し、**売上要因分解 → AI考察 → 施策ボード → 効果検証 → 月次レポート** までを1つの導線で繋ぎます。BPaaS伴走（人間レビュー / 月次会議）が前提に組み込まれている点も差別化要素です。

### Q8. 「カスタマイズはどこまで効きますか？」

A. **MVP段階は固定UI** です。Phase 4 以降のロードマップで、テナント別のKPI定義 / レポートテンプレート / 施策カテゴリの調整を予定しています。商談後にお客さまの要望に応じて優先順位を再設計します。

### Q9. 「セキュリティは？認証は？」

A. **現MVPは認証なし、データはブラウザ内 localStorage のみ**。プロトタイプ段階のため意図的に未実装です。Phase 4 で **Clerk / Supabase Auth** による組織アカウント / 役割管理 / 操作ログを実装する計画です。本番導入時の SLA / SOC2 等は別途ご相談ください。

### Q10. 「いつから使えるんですか？」

A. **現状は静的MVPプロトタイプ**で、CSV取込・売上要因分解・施策ボード・月次レポートのレイアウトはご覧の通り動作します。AI実接続 / 永続化 / 認証 / API連携 / PDF出力は **6〜12ヶ月の段階的実装** を想定。CSVベースの月次BPaaS伴走サービスとしての開始は **より早期に可能** です。
